# Logos Bible Study Chatbot — Detailed Architecture

Vachan Study (Logos Bible Study Chatbot) is a highly interactive, full-stack scripture study application. It combines a modern Next.js frontend with a retrieval-first Python/FastAPI backend. The system is designed for conversational, multilingual, and voice-first interaction, using hybrid vector search to ground AI responses directly in scripture datasets.

---

## 🏗️ System Architecture Overview

### 1. Frontend Layer (Next.js + React)
The frontend provides a rich, responsive UI utilizing **Tailwind CSS** and **Framer Motion** for animations.

**Key Components (`src/components/Workspace.tsx`):**
- **Chat Interface:** Real-time streaming response UI using Server-Sent Events (SSE). Includes status updates (e.g., "Searching Scripture...", "Translating...").
- **Voice-First Input (STT):** Uses the `MediaRecorder` API to capture high-quality audio (`audio/webm;codecs=opus`), negotiates the best MIME type, and posts the audio blob to the backend for AI transcription.
- **Hands-Free Output (TTS):** Employs a custom "Autoplay Whitelisting" mechanism to defeat strict browser media blockers. Audio is streamed natively from the backend as pure Base64 JSON and loaded directly into a DOM-attached `<audio>` element.
- **Scripture Panel:** Displays the selected book (e.g., Genesis). Automatically highlights specific verses referenced by the AI during a chat.
- **Suggestion Chips:** Dynamic contextual follow-up questions generated by the AI based on the conversation context.

### 2. Backend Layer (Python + FastAPI)
The backend manages all intelligence, orchestration, retrieval, and translation logic.

**Key Endpoints (`backend/api/index.py`):**
- `POST /api/chat`: The core intelligence loop. Processes history, runs language detection, performs hybrid search, formats RAG context, queries Gemini, and streams the response back via SSE.
- `POST /api/transcribe`: Receives raw audio blobs from the frontend, uses Gemini 2.5 Flash to generate highly accurate multilingual transcripts, and returns the parsed text.
- `POST /api/tts`: Receives text payload, auto-detects language (e.g., `ml` vs `en`), utilizes `gTTS` to generate raw MP3 streams, encodes the binary to a Base64 JSON payload, and returns it to the client to avoid HTTP chunk corruption.
- `GET/DELETE /api/history/{book}`: Manages user conversation history directly in MongoDB.

### 3. Data & Storage Layer (MongoDB Atlas)
The legacy local FAISS vector indices have been completely replaced with **MongoDB Atlas Vector Search** for scalable, cloud-native Hybrid Search.

**Collections:**
- **`qa_dataset`**: The core RAG knowledge base containing ~18,000 embedded Q&A pairs. 
  - Schema includes: `book`, `chapter`, `verse`, `question`, `answer`, `lang_code` (`en`, `ml`, `hi`), and `embedding`.
  - The `embedding` field contains exactly 768-dimension vectors (truncated from Gemini's native 3072 dimensions) using Matryoshka Representation Learning to save 75% storage space.
- **`chat_history`**: Persistent user conversation logs, organized by `session_id` and `book`.

**Search Indices:**
- `vector_index`: Performs semantic similarity search on the `embedding` field.
- `text_index`: Performs BM25 keyword matching on the `question` field.

## 🧠 Intelligence Flow: Hybrid RAG Pipeline

When a user asks a question (e.g., "What did God create in the beginning?"):

1. **Language Detection & Translation:**
   - The user's query is analyzed using `langdetect`.
   - If the query is non-English (e.g., Malayalam), the system automatically translates the query to English to maximize search recall against the core dataset.

2. **Hybrid Search (`backend/services/rag.py`):**
   - The query is embedded into a 768-dimensional vector using Google's `text-embedding-004` model.
   - A concurrent MongoDB query executes both:
     - **Vector Search** (Semantic similarity)
     - **Text Search** (BM25 Keyword match)
   - Results from both algorithms are merged.

3. **Reciprocal Rank Fusion & Cross-Encoder Re-Ranking:**
   - The merged results are re-ranked using a Cross-Encoder (`cross-encoder/ms-marco-MiniLM-L-6-v2`).
   - The system establishes a strict confidence threshold. If a document scores high confidence (`>= 0.15`), it is classified as a "Direct Hit".

4. **Generation & Fallback Formulation:**
   - **Direct Hit:** The system instantly returns the pre-approved, dataset-backed answer without spending tokens on LLM generation, ensuring 100% theological accuracy.
   - **Low Confidence:** The system falls back to a Gemini AI prompt, feeding it the retrieved scripture context and asking it to generate a faithful answer.

5. **Native Translation Return:**
   - If the original query was non-English, the final English answer is translated back into the user's native tongue before being streamed to the frontend.
